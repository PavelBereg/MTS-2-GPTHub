# Долгосрочная память (Memory 2.0)

MTS AI Assistant не ограничен текущей сессией браузера. Он обладает **Долгосрочной памятью**, которая может извлекать и сохранять знания о пользователе. Эта система работает как автономный фоновый процесс.

Архитектура механизма памяти строится на использовании векторной базы данных **Qdrant** и эмбеддингах **bge-m3** (через MWS API).

## Двухуровневая система Qdrant

Используются две разные коллекции:

1. **`user_memory`**: Сохраняет сырые фрагменты контекста (предыдущих диалогов). Служит для семантического поиска истории.
2. **`user_facts`**: Структурированные факты о пользователе (имя, должность, проекты, цели). Избавлены от шума длинных сообщений.

## Извлечение Фактов (Fact Extraction)

Когда пользователь отправляет сообщение, бэкенд не только отвечает, но и запускает **неблокирующую фоновую задачу** `extract_and_save_facts`. 

Процесс извлечения (см. `app/memory/fact_extractor.py`):
1. Отправляем текст запроса в быструю модель `llama-3.1-8b-instruct`.
2. Модель (с помощью специального промпта для классификации) выделяет атомарные факты и возвращает JSON.
3. Разрешенные типы фактов: `name`, `preference`, `project`, `role`, `context`, `goal`, `general`.
4. Извлечённые факты конвертируются в эмбеддинги и записываются в `user_facts`.

> [!WARNING]
> Система фактов игнорирует общие знания (научные темы, погоду) и концентрируется только на персонализированной информации о самом пользователе.

## Использование Памяти

Перед каждым вызовом LLM в диалоге, вызывается функция `_get_memory_context`. Её задача:

- Сделать семантический поиск по `user_memory`.
- Получить весь профиль фактов пользователя.
- Найти релевантные факты по текущему запросу в `user_facts` (дедуплицируя из общего профиля).

Собранная информация агрегируется в единый текстовый текстовый блок (см. `build_memory_context`) и прозрачно "вшивается" в System Prompt перед отправкой в LLM (MWS API), таким образом модель всегда "помнит" кто перед ней и о чём до этого шел разговор, даже при очистке истории чата в UI.
